# 智能判题系统需求规格说明书

文档版本：V0.1

更新日期：2026-03-20

## 1. 文档概述

### 1.1 编写目的
本说明书用于明确智能判题系统的建设目标、功能范围、技术方案、质量要求、实施计划及人员分工，为项目开发、测试、验收与交付提供统一依据。

### 1.2 项目背景
本项目拟开发一套基于桌面端的智能判题系统，面向拍照采集答题内容、自动识别与智能评分场景，重点支持填空题、简答题及计算题等题型的识别、正误判断与分值计算。系统支持在未提供标准答案时进行自主判断，在提供标准答案后进行进一步校准与辅助判定，并兼顾过程分计算、公式规范显示及低成本运行需求。系统建设周期为 **7 个工作日**，由 **5 名成员** 协同完成。

### 1.3 技术栈说明
- 开发语言：Python  
- 桌面框架：tkinter  
- 网络请求：requests  
- 图像处理：Pillow  
- 智能判题能力：外部智能判题服务接口（供应商待定）  

### 1.4 建设目标
本项目建设目标如下：
- 完成桌面端智能判题系统的基础框架搭建与功能实现；
- 打通拍照采集、图像识别、判分、结果反馈的完整业务流程；
- 支持填空题、简答题及计算题等主要题型的识别与正误判断；
- 支持在提供标准答案后进行判分校准，并兼容多种标准答案写法及语义近似表达；
- 解决图片预览压缩、提交按钮显示异常、LaTeX 公式乱码等关键问题；
- 在满足识别效果和判分准确性的前提下，优先采用低成本、国产化的外部智能判题能力；
- 形成可直接运行的软件版本及配套使用文档、部署文档。

## 2. 项目范围

### 2.1 建设范围
本项目建设范围包括以下内容：
- 智能判题桌面端界面设计与实现；
- 拍照采集、拍照上传与图像预处理；
- 调用大模型接口完成题目识别与智能评分；
- 填空题、简答题、计算题等题型的识别、判分与过程分计算逻辑实现；
- 基于标准答案的校准判分、等效答案识别与语义辅助匹配；
- LaTeX 公式格式化与无乱码显示处理；
- 防重复提交、异常提示等交互优化；
- 系统测试、用户文档与部署文档编制。

### 2.2 非建设范围
本阶段不包含以下内容：
- 移动端 App 开发；
- 本地图片文件手工导入作为正式业务输入方式；
- 本地题库批量导入与离线批量阅卷；
- 大规模分布式部署与云原生改造；
- 独立后台管理系统建设；
- 与第三方教务系统的深度集成。

## 3. 功能需求

### 3.1 界面展示与交互模块
该模块负责系统桌面端整体界面布局、页面交互及用户操作反馈。

**功能要求：**
- 提供清晰、规范的桌面端操作界面；
- 支持上传按钮、提交按钮、结果展示区域等核心控件的稳定显示；
- 修复提交按钮显示异常问题，确保在不同窗口尺寸和交互状态下均可正常使用；
- 优化界面交互体验，使操作流程简洁、直观、可重复执行。

### 3.2 拍照采集与上传模块
该模块负责通过拍照方式采集答题内容，并完成上传、展示及预处理。

**功能要求：**
- 支持通过摄像头拍照采集答题内容并上传；
- 系统仅支持拍照上传场景，不支持本地图片文件手工导入作为正式业务输入方式；
- 支持图片预览，确保展示效果清晰、完整；
- 针对图片预览压缩问题进行优化，避免因压缩导致内容失真或影响识别效果；
- 对不符合格式要求的图片进行提示与拦截。

### 3.3 智能判题与题型识别模块
该模块负责调用外部智能判题服务接口，实现题型识别、内容识别、正误判断与评分。

**功能要求：**
- 完成与外部智能判题服务接口的 API 对接；
- 支持填空题、简答题及计算题等主要题型的精准识别与分类；
- 在未提供标准答案时，支持基于题目内容与作答内容进行自主判断，并输出正误结论；
- 支持过程分计算，并根据题目作答情况输出评分结果；
- 对接口异常、超时、返回格式异常等情况提供可追踪的错误提示。

### 3.4 标准答案校准与语义匹配模块
该模块负责在提供标准答案后，对系统判定结果进行进一步校准，并处理多写法、多变体答案及语义近似情况。

**功能要求：**
- 对于填空题，在给定标准答案时，支持基于标准答案对原始判定结果进行进一步调整；
- 支持多个标准答案变体的配置与识别，兼容同义表达、格式差异、符号差异及常见等效写法；
- 对于简答题，在给定标准答案时，支持基于关键词、关键要点、语义相似度或向量化表征进行辅助匹配与校准判分；
- 当标准答案存在多种合理写法时，系统应支持等效答案归并与容错判定；
- 系统输出结果应能够区分“自主判断结果”与“基于标准答案校准后的结果”。

### 3.5 公式处理与显示模块
该模块负责题目及评分结果中的公式规范显示。

**功能要求：**
- 支持 LaTeX 公式内容的格式化处理；
- 解决公式乱码问题，确保数学表达式可正常展示；
- 保证公式在界面中的可读性与排版一致性；
- 在题目识别、评分结果展示等环节中保持公式前后一致。

### 3.6 提交控制与异常防护模块
该模块负责控制重复提交、异常操作及流程稳定性。

**功能要求：**
- 增加防重复提交机制，避免因重复点击导致重复请求；
- 在提交中、完成后、失败时提供明确状态提示；
- 对未拍照上传、网络异常、识别失败、判分失败等情况进行友好提示；
- 确保系统主流程在异常情况下可恢复、可重试。

### 3.7 测试与验收支持模块
该模块负责系统测试、缺陷跟踪及验收支持。

**功能要求：**
- 覆盖主要业务流程的功能测试；
- 对填空题自主判定、简答题自主判定、标准答案校准、等效答案识别及过程分计算进行重点验证；
- 记录缺陷问题并按优先级推动修复；
- 对界面交互、评分结果、异常处理等进行重点验证；
- 确保系统功能通过率达到 **98% 及以上**。

### 3.8 文档与交付模块
该模块负责项目文档整理及最终交付。

**功能要求：**
- 编写用户操作手册；
- 编写部署说明文档与依赖清单；
- 整理可直接运行的完整代码；
- 完成交付材料归档，确保软件与文档一致。

## 4. 非功能需求

### 4.1 易用性要求
- 系统操作流程应清晰明确，普通用户可快速上手；
- 核心功能按钮位置合理，提示信息明确；
- 异常提示应具有可理解性，避免模糊表述。

### 4.2 稳定性要求
- 系统在连续使用过程中应保持界面稳定，不出现主要控件丢失或卡死现象；
- 对重复提交、接口超时、异常返回等场景具备基本容错能力。

### 4.3 兼容性要求
- 系统应在既定 Python 运行环境下稳定运行；
- 图像处理、公式显示及网络调用应与当前项目依赖保持兼容。

### 4.4 可维护性要求
- 模块划分清晰，便于后续功能扩展与问题定位；
- 依赖清单、部署步骤及用户说明应完整可复用。

### 4.5 成本预算要求
- 本项目运行期资源预算原则上以供应商**正式 API 商业计费方案**为基线，不以试用额度、赠送额度、免费额度或编码订阅额度作为正式运行成本承诺依据；
- 本版规格说明书为便于评审与测算，采用“**1000 份试卷 / 1 万次模型调用**”作为预算基线场景；
- 鉴于部分平台公开价格页存在动态渲染或控制台展示差异，本版预算测算采用**已调研的国内主流平台公开商业计费口径**作为可核验基线；项目正式采购确定后，应以签约价格或控制台实时价格替换本节测算值；
- 在满足识别效果、判分准确率及合规要求的前提下，应优先选择国产化、低成本的外部智能判题服务方案；
- 预算管理应预留异常重试、接口波动、人工复核及后续模型切换的缓冲空间，不应仅按理论最低调用成本配置。

### 4.6 预算测算基线

#### 4.6.1 测算假设
为保证测算口径清晰统一，本项目暂按以下假设进行预算估算：

- 业务规模：**1000 份试卷**；
- 调用规模：**1 万次模型调用**，折合平均每份试卷约 **10 次调用**；
- 调用结构：约 **40%** 为拍照识别、题面理解等多模态调用，约 **60%** 为正误判断、标准答案校准、过程分解释与结果生成调用；
- 多模态测算口径：按国内主流商业多模态接口的公开中档价格估算；
- 文本判分测算口径：按国内主流商业文本推理接口的公开中档价格估算；
- 多模态单次调用假设：图片/视频输入约 **3000 Tokens**、文本输入约 **1000 Tokens**、文本输出约 **1200 Tokens**；
- 文本单次调用假设：文本输入约 **2000 Tokens**、文本输出约 **1500 Tokens**。

#### 4.6.2 测算结果

| 调用类型 | 参考口径 | 调用次数 | 单次测算成本 | 小计 |
|---|---|---:|---:|---:|
| 多模态识题与图文理解 | 国内主流商业多模态接口公开价格口径 | 4,000 次 | 约 0.02694 元/次 | 约 107.76 元 |
| 文本判分与结果生成 | 国内主流商业文本推理接口公开价格口径 | 6,000 次 | 约 0.00880 元/次 | 约 52.80 元 |
| **合计** | - | **10,000 次** | - | **约 160.56 元** |

考虑到项目在实际运行中可能存在重复提交、异常重试、网络波动、长文本输出增加、模型切换及人工复核补充调用等情况，建议在基线测算结果上增加 **20%** 预算缓冲。

- 基线调用成本：**约 160.56 元**；
- 加计 20% 缓冲后：**约 192.67 元**；
- 规格说明书建议预算口径：**按 200 元 / 1000 份试卷** 作为阶段性估算值；
- 折算单份试卷平均模型成本：**约 0.19~0.20 元 / 份**。

#### 4.6.3 预算说明
- 本测算值主要用于立项讨论、方案比选及资源预留，不构成最终采购结算依据；
- 项目正式确定供应商及服务方案后，应依据对应正式价格重新测算；
- 若项目以“尽量免费”为阶段性目标，可在前期验证阶段优先使用国产平台免费额度或低成本方案，但不得将其视为正式运行期的长期预算基础；
- 若项目进入批量化运行阶段，应进一步按“日调用量、峰值并发、异常重试率、人工抽检比例”进行扩展预算评估。

### 4.7 免费额度与 Coding Plan 额度使用边界及风险

#### 4.7.1 免费额度的使用边界与风险
各平台提供的免费额度、赠送额度或新用户优惠，原则上属于试用或促销性质资源，可用于前期验证、功能联调和阶段性成本测算，但不应作为正式运行期的长期预算前提。

当前已调研的国内主流平台公开规则显示，免费额度或新用户优惠通常具有以下共性特征：

- 仅在特定区域、特定账户资格或特定服务范围内有效；
- 普遍存在 **30~90 天** 左右的有效期或 **1 个月左右** 的促销周期；
- 主要覆盖标准实时推理费用，不覆盖批量处理、上下文缓存、模型调优、模型部署或定制化能力相关费用。

基于上述规则，免费额度用于本项目时的风险主要表现为：

1. **服务连续性风险**：免费额度存在有效期、适用范围和额度上限，资源到期、耗尽或平台政策调整后，项目将无法继续按原预算运行；
2. **预算失真风险**：若以免费额度覆盖开发中后期或试运行阶段，会低估正式上线后的真实成本；
3. **自动扣费风险**：部分平台在免费额度耗尽后，若未开启用量停止或预算控制机制，后续调用将自动转为按量计费，并可能导致账户扣费或欠费；
4. **适用范围风险**：免费额度通常只覆盖标准实时推理，不覆盖缓存、批量处理、微调、部署等扩展能力，因此不适合作为完整项目成本模型。

就现有公开证据而言，免费额度用于本项目的主要风险应认定为**服务连续性风险、预算管理风险及合同稳定性风险**，不宜直接表述为一般意义上的法律禁止。

因此，免费额度在本项目中的定位应明确为：

> **仅可作为开发测试、功能验证及阶段性估算参考，不作为正式运行期预算承诺依据。**

#### 4.7.2 Coding Plan 额度的使用边界与风险
Coding Plan 本质上是面向 AI 编程工具的订阅型资源，更适合代码生成、调试、重构与开发提效场景，不应默认等同于可供业务系统直接调用的正式模型服务能力。

当前已调研的国内主流 Coding Plan 产品公开规则显示：

- 部分平台已明确限定其额度**仅在 AI 编程工具场景中生效，不可作为普通业务系统 API 调用资源**；
- 其余平台虽未一律以绝对禁止方式表述，但公开页面与帮助文档均将其定位于 AI 编程工具接入场景，并按编程请求额度或时间窗口进行管理；
- 该类资源整体上更适用于开发辅助与工程提效，而非正式业务运行期的推理入口。

基于上述规则，Coding Plan 额度用于本项目时的风险主要表现为：

1. **平台规则违约风险**：对于已明确限制“不可用于 API 调用”的平台，如将 Coding Plan 额度接入本项目运行时后端，存在违反平台规则的风险；
2. **账号停用风险**：一旦被识别为非编程工具调用、批量化调用或异常调用特征，可能触发限流、订阅失效或账号封禁；
3. **合同适配风险**：即使公开页面未直接表述为“法律禁止”，其产品定位仍明显偏向开发辅助工具，若用于正式业务运行，将存在用途不匹配与合同解释风险；
4. **运行稳定性风险**：Coding Plan 的额度体系通常围绕“请求次数/时间窗口”设计，更适合交互式开发，不适合作为高稳定性、可追踪、可审计的生产运行预算基础。

因此，Coding Plan 的风险表述应以**平台规则限制、账号治理风险与合同适配风险**为主；对于缺乏明确禁止条款、仅体现产品定位的平台，不宜直接上升为一般意义上的法律禁止。

因此，Coding Plan 额度在本项目中的定位应明确为：

> **可用于开发阶段的代码辅助与工程提效，不可作为本项目正式运行期的后端模型调用预算基础。**

#### 4.7.3 本项目的合规预算结论
综合预算测算与外部资源使用边界，本项目应采用如下原则：

- 正式业务运行预算以**商业 API 正式计费**为准；
- 免费额度、赠送额度仅用于前期测试与验证；
- Coding Plan 仅用于开发提效，不纳入业务运行成本模型；
- 任何存在明确使用范围限制的平台资源，均不得作为项目运行时推理入口。

## 5. 实施计划

### 5.1 第一阶段：需求明确与环境准备
在项目初期，项目组需明确核心需求、技术栈及质量标准，前后端开发人员完成需求拆解并确认模块优先级，文档运维人员同步梳理第三方依赖清单并搭建开发环境。

### 5.2 第二阶段：方案设计与评审
前端开发人员输出界面布局设计稿，制定图片压缩与按钮显示异常的解决方案；后端开发人员梳理 API 调用逻辑、公式格式化逻辑并绘制流程图；项目组完成集中评审并确认最终实施方案。

### 5.3 第三阶段：功能开发与联调
前端完成界面框架搭建及图片上传预览模块开发；后端完成 API 集成、公式处理与防重复提交功能开发；随后开展系统整合与初步自测。

### 5.4 第四阶段：测试优化与问题修复
测试工程师执行功能测试并记录 Bug，开发人员依据优先级进行修复，同时优化界面交互体验，确保系统整体质量满足验收要求。

### 5.5 第五阶段：文档编制与最终交付
