## 1. 文档概述

### 1.1 编写目的
本说明书用于明确智能判题系统的建设目标、功能范围、技术方案、质量要求、实施计划及人员分工，为项目开发、测试、验收与交付提供统一依据。

### 1.2 项目背景
本项目拟开发一套基于 **Web 应用（B/S 架构）** 的智能判题系统，面向拍照采集答题内容、自动识别与智能评分场景，重点支持填空题、简答题及计算题等题型的识别、正误判断与分值计算。系统支持在未提供标准答案时进行自主判断，在提供标准答案后进行进一步校准与辅助判定，并兼顾过程分计算、公式规范显示及低成本运行需求。系统建设周期为 **7 个工作日**，由 **5 名成员** 协同完成。

> **V0.2 架构调整说明**：原 V0.1 版本采用 tkinter 桌面端架构。经评估，桌面端架构在以下方面无法满足项目实际需求：
> - **部署成本**：全校 1 万人规模下，逐台安装 Python 运行环境不具备可操作性；
> - **密钥安全**：API Key 无法安全分发至每台客户端；
> - **题库共享**：桌面端无集中存储能力，题库无法在用户间共享；
> - **调用管控**：无法实现调用量统计、用户配额与限流。
>
> 因此，本版将系统架构调整为 Web 应用（B/S），服务端集中部署，用户通过浏览器访问，解决上述问题。

### 1.3 技术栈说明

| 层 | 技术 | 说明 |
|----|------|------|
| 开发语言 | Python 3.10+ | 全栈统一语言 |
| 后端框架 | FastAPI | 异步 Web 框架，自动生成 API 文档 |
| 数据库 | SQLite（开发/小规模部署）→ PostgreSQL（高并发） | SQLite 零配置；后续按需切换 |
| ORM | SQLModel | Pydantic + SQLAlchemy 合一，与 FastAPI 深度集成 |
| 前端 | 原生 HTML + CSS + JavaScript | 无需额外前端框架，团队直接上手 |
| 网络请求（服务端） | httpx（异步） | 替代 requests，不阻塞服务器 |
| 图像处理 | Pillow | 图片压缩与预处理 |
| 文档解析 | python-docx + pdfplumber | 解析 Word/PDF 中的题目与标准答案 |
| 模板引擎 | Jinja2 | 服务端渲染 HTML 页面（可选） |
| 智能判题能力 | 外部 LLM 服务接口（OpenAI 兼容） | 供应商待定，优先国产低成本方案 |

#### 依赖清单

```bash
pip install fastapi uvicorn sqlmodel httpx python-multipart python-docx pdfplumber pillow jinja2
```

### 1.4 建设目标
本项目建设目标如下：
- 完成 Web 端智能判题系统的基础框架搭建与功能实现；
- 打通拍照采集、图像识别、判分、结果反馈的完整业务流程；
- 支持填空题、简答题及计算题等主要题型的识别与正误判断；
- 支持在提供标准答案后进行判分校准，并兼容多种标准答案写法及语义近似表达；
- 实现题库管理功能，支持通过 Word/PDF 文档批量导入题目与标准答案；
- 实现结构化过程分输出，计算题逐步展示评分依据，具备可解释性与可交互性；
- 实现服务端集中部署，全校用户通过浏览器访问，API Key 仅服务端持有；
- 在满足识别效果和判分准确性的前提下，优先采用低成本、国产化的外部智能判题能力；
- 形成可直接运行的软件版本及配套使用文档、部署文档。

## 2. 项目范围

### 2.1 建设范围
本项目建设范围包括以下内容：
- 智能判题 Web 端界面设计与实现（判题页 + 题库管理页）；
- 基于浏览器的拍照采集、图片上传与图像预处理；
- 后端 RESTful API 设计与实现（FastAPI）；
- 调用大模型接口完成题目识别与智能评分；
- 填空题、简答题、计算题等题型的识别、判分与过程分计算逻辑实现；
- 基于标准答案的校准判分、等效答案识别与语义辅助匹配；
- 题库管理：支持 Word/PDF 文档上传、自动解析题目-答案对、题库增删查；
- 判题结果结构化输出（JSON 格式），支持过程分步骤级展示；
- LaTeX 公式格式化与规范显示处理；
- 防重复提交、异常提示等交互优化；
- 服务端部署方案（单机 + 可选 Nginx 反向代理）；
- 系统测试、用户文档与部署文档编制。

### 2.2 非建设范围
本阶段不包含以下内容：
- 独立移动端 App 开发（但 Web 端支持手机浏览器拍照上传）；
- 大规模分布式部署与云原生改造；
- 独立后台管理系统（管理功能集成在题库管理页中）；
- 与第三方教务系统的深度集成；
- 用户登录与权限管理（首版不做，后续按需迭代）。

### 3.1 整体架构

```
用户（浏览器）
  │
  │  HTTP / HTTPS
  ▼
┌──────────────────────────────────┐
│  FastAPI 服务端                   │
│  ├── 路由层（routers/）           │  ← API 入口
│  ├── 服务层（services/）          │  ← 业务逻辑

└──────────────────────────────────┘
  │
  │  httpx（异步）

```

### 3.3 API 路由设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/judge` | 上传图片 + 可选答案 → 判题结果 |
| POST | `/api/judge/with-bank` | 上传图片 → 自动匹配题库答案 → 判题 |
| GET | `/api/results` | 查询历史判题记录 |
| POST | `/api/upload/document` | 上传 Word/PDF → 解析入库 |
| GET | `/api/questions` | 题库列表（支持分页与搜索） |
| POST | `/api/questions` | 手动添加题目 |
| DELETE | `/api/questions/{id}` | 删除题目 |

### 3.4 数据模型

#### 题库模型（Question）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int（主键） | 自增 ID |
| content | str | 题面文本 |
| question_type | str | 题型：fill_blank / short_answer / calculation |
| standard_answer | str（可选） | 标准答案 |
| source_file | str（可选） | 来源文件名 |
| created_at | datetime | 创建时间 |

#### 判题记录模型（JudgeResult）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int（主键） | 自增 ID |
| image_path | str | 上传图片路径 |
| question_type | str | 识别出的题型 |
| recognized_content | str | OCR 识别内容 |
| judgment | str | 正确 / 错误 / 部分正确 |
| score | float | 得分 |
| total_score | float | 满分 |
| explanation | str | 详细解析 |
| step_scores | str（可选） | 过程分 JSON（计算题） |
| matched_question_id | int（可选） | 匹配到的题库题目 ID |
| created_at | datetime | 创建时间 |

### 3.5 判题输出格式

系统要求 LLM 以 **结构化 JSON** 返回判题结果，替代 V0.1 版本的自由文本输出，以实现前端结构化渲染和数据持久化：

```json
{
  "question_type": "calculation",
  "recognized_content": "求 ∫x²dx 的不定积分，学生作答：x³/3",
  "judgment": "partial",
  "score": 8,
  "total_score": 10,
  "explanation": "积分计算正确，但缺少积分常数 C",
  "steps": [
    {"step": "被积函数识别：x²", "correct": true, "score": 2, "comment": "正确识别"},
    {"step": "积分计算：x³/3", "correct": true, "score": 4, "comment": "幂函数积分公式正确"},
    {"step": "积分常数 +C", "correct": false, "score": 0, "comment": "缺少积分常数"},
    {"step": "最终结果", "correct": false, "score": 2, "comment": "结果部分正确，扣 2 分"}
  ]
}
```

## 4. 功能需求

### 4.1 界面展示与交互模块
该模块负责系统 Web 端整体界面布局、页面交互及用户操作反馈。

**功能要求：**
- 提供清晰、规范的 Web 端操作界面，包含判题页与题库管理页；
- 支持上传按钮、提交按钮、结果展示区域等核心控件的稳定显示；
- 界面自适应不同屏幕尺寸，支持 PC 浏览器与手机浏览器访问；
- 优化界面交互体验，使操作流程简洁、直观、可重复执行。

### 4.2 拍照采集与上传模块
该模块负责通过拍照或选择图片方式采集答题内容，并完成上传、展示及预处理。

**功能要求：**
- 支持通过浏览器调用摄像头拍照（`<input type="file" capture="camera">`）或从相册选择图片上传；
- 支持 PNG / JPG / JPEG / BMP 格式；
- 支持图片预览，确保展示效果清晰、完整；
- 在上传前对图片进行必要的压缩与格式校验；
- 对不符合格式要求的文件进行提示与拦截。

### 4.3 智能判题与题型识别模块
该模块负责调用外部智能判题服务接口，实现题型识别、内容识别、正误判断与评分。

**功能要求：**
- 完成与外部智能判题服务接口（OpenAI 兼容 Chat Completions）的 API 对接；
- 支持填空题、简答题及计算题等主要题型的精准识别与分类；
- 在未提供标准答案时，支持基于题目内容与作答内容进行自主判断，并输出正误结论；
- 支持过程分计算，计算题需逐步拆分评分，输出每一步的得分与点评（结构化 JSON）；
- LLM 调用采用异步方式（httpx），不阻塞服务器处理其他请求；
- 对接口异常、超时、返回格式异常等情况提供可追踪的错误提示。

### 4.4 标准答案校准与语义匹配模块
该模块负责在提供标准答案后，对系统判定结果进行进一步校准，并处理多写法、多变体答案及语义近似情况。

**功能要求：**
- 对于填空题，在给定标准答案时，支持基于标准答案对原始判定结果进行进一步调整；
- 支持多个标准答案变体的配置与识别，兼容同义表达、格式差异、符号差异及常见等效写法；
- 对于简答题，在给定标准答案时，支持基于关键词、关键要点、语义相似度进行辅助匹配与校准判分；
- 当标准答案存在多种合理写法时，系统应支持等效答案归并与容错判定；
- 系统输出结果应能够区分"自主判断结果"与"基于标准答案校准后的结果"。

### 4.5 题库管理与文档解析模块
该模块负责题库的构建、维护与文档自动解析，是本次架构调整新增的核心模块。

**功能要求：**
- 支持老师上传 Word（.docx）或 PDF 文档，系统自动解析出题目与标准答案的配对关系；
- 支持常见的题目编号格式（1. / 1、/ （1）/ 第一题 等）；
- 支持"题目在前、答案在后"和"题目区与答案区分离"两种文档结构；
- 解析结果在页面展示供老师确认后入库，避免解析错误直接入库；
- 题库支持增删查，老师可手动添加、修改或删除题目与答案；
- 题库具备可增长性，支持多次上传、逐步积累；
- 判题时支持自动匹配题库中的对应题目，找到标准答案后辅助判分。

### 4.6 公式处理与显示模块
该模块负责题目及评分结果中的公式规范显示。

**功能要求：**
- 支持 LaTeX 公式内容的格式化处理；
- 在 Web 前端中使用 KaTeX 或 MathJax 渲染数学公式，彻底解决乱码问题；
- 保证公式在界面中的可读性与排版一致性；
- 在题目识别、评分结果展示等环节中保持公式前后一致。

### 4.7 提交控制与异常防护模块
该模块负责控制重复提交、异常操作及流程稳定性。

**功能要求：**
- 增加防重复提交机制，提交期间禁用按钮，完成后自动恢复；
- 在提交中、完成后、失败时提供明确状态提示；
- 对未上传图片、网络异常、识别失败、判分失败等情况进行友好提示；
- 确保系统主流程在异常情况下可恢复、可重试。

### 4.8 测试与验收支持模块
该模块负责系统测试、缺陷跟踪及验收支持。

**功能要求：**
- 覆盖主要业务流程的功能测试（API 级 + 端到端）；
- 对填空题自主判定、简答题自主判定、标准答案校准、等效答案识别、过程分计算及题库管理进行重点验证；
- 记录缺陷问题并按优先级推动修复；
- 确保系统功能通过率达到 **98% 及以上**。

### 4.9 文档与交付模块
该模块负责项目文档整理及最终交付。

**功能要求：**
- 编写用户操作手册；
- 编写部署说明文档与依赖清单；
- 整理可直接运行的完整代码；
- 编写环境变量配置说明（`.env.example`）；
- 完成交付材料归档，确保软件与文档一致。

## 5. 非功能需求

### 5.1 易用性要求
- 系统操作流程应清晰明确，普通用户可快速上手；
- 用户无需安装任何软件，通过浏览器访问即可使用；
- 核心功能按钮位置合理，提示信息明确；
- 异常提示应具有可理解性，避免模糊表述。

### 5.2 稳定性要求
- 系统在连续使用过程中应保持运行稳定；
- 后端采用异步框架（FastAPI + httpx），避免单次 LLM 调用阻塞其他用户请求；
- 对重复提交、接口超时、异常返回等场景具备基本容错能力。

### 5.3 兼容性要求
- 系统后端应在 Python 3.10+ 环境下稳定运行；
- 前端应兼容主流浏览器（Chrome、Edge、Firefox、Safari）；
- 前端应支持 PC 和手机浏览器访问（响应式布局）。

### 5.4 安全性要求
- API Key 仅在服务端 `.env` 文件中配置，**严禁** 暴露在前端代码或客户端；
- `.env` 文件必须加入 `.gitignore`，不得提交至版本控制；
- 文件上传接口应校验文件类型与大小，防止恶意文件上传。

### 5.5 可维护性要求
- 后端代码按路由层（routers）、服务层（services）、数据层（models）、工具层（utils）清晰分层；
- 依赖清单、部署步骤及用户说明应完整可复用；
- 数据库采用 SQLite 单文件，备份仅需复制一个文件。

### 5.6 部署要求

#### 开发环境

```bash
pip install -r requirements.txt
uvicorn backend.main:app --reload --host 0.0.0.0 --port 8000
```

浏览器打开 `http://localhost:8000` 即可使用。

#### 生产环境（校内部署）

```bash
# 最简方案：一台校内服务器
uvicorn backend.main:app --host 0.0.0.0 --port 80 --workers 4
```

- 全校用户通过校园网访问 `http://<服务器IP>`；
- API Key 仅在服务器 `.env` 中配置，不暴露给任何用户；
- SQLite 单文件数据库，备份只需复制一个文件。

### 5.7 成本预算要求
- 本项目运行期资源预算原则上以供应商**正式 API 商业计费方案**为基线，不以试用额度、赠送额度、免费额度或编码订阅额度作为正式运行成本承诺依据；
- 本版规格说明书为便于评审与测算，采用"**1000 份试卷 / 1 万次模型调用**"作为预算基线场景；
- 鉴于部分平台公开价格页存在动态渲染或控制台展示差异，本版预算测算采用**已调研的国内主流平台公开商业计费口径**作为可核验基线；项目正式采购确定后，应以签约价格或控制台实时价格替换本节测算值；
- 在满足识别效果、判分准确率及合规要求的前提下，应优先选择国产化、低成本的外部智能判题服务方案；
- 预算管理应预留异常重试、接口波动、人工复核及后续模型切换的缓冲空间，不应仅按理论最低调用成本配置。

### 5.8 预算测算基线

#### 5.8.1 测算假设
为保证测算口径清晰统一，本项目暂按以下假设进行预算估算：

- 业务规模：**1000 份试卷**；
- 调用规模：**1 万次模型调用**，折合平均每份试卷约 **10 次调用**；
- 调用结构：约 **40%** 为拍照识别、题面理解等多模态调用，约 **60%** 为正误判断、标准答案校准、过程分解释与结果生成调用；
- 多模态测算口径：按国内主流商业多模态接口的公开中档价格估算；
- 文本判分测算口径：按国内主流商业文本推理接口的公开中档价格估算；
- 多模态单次调用假设：图片/视频输入约 **3000 Tokens**、文本输入约 **1000 Tokens**、文本输出约 **1200 Tokens**；
- 文本单次调用假设：文本输入约 **2000 Tokens**、文本输出约 **1500 Tokens**。

#### 5.8.2 测算结果

| 调用类型 | 参考口径 | 调用次数 | 单次测算成本 | 小计 |
|---|---|---:|---:|---:|
| 多模态识题与图文理解 | 国内主流商业多模态接口公开价格口径 | 4,000 次 | 约 0.02694 元/次 | 约 107.76 元 |
| 文本判分与结果生成 | 国内主流商业文本推理接口公开价格口径 | 6,000 次 | 约 0.00880 元/次 | 约 52.80 元 |
| **合计** | - | **10,000 次** | - | **约 160.56 元** |

考虑到项目在实际运行中可能存在重复提交、异常重试、网络波动、长文本输出增加、模型切换及人工复核补充调用等情况，建议在基线测算结果上增加 **20%** 预算缓冲。

- 基线调用成本：**约 160.56 元**；
- 加计 20% 缓冲后：**约 192.67 元**；
- 规格说明书建议预算口径：**按 200 元 / 1000 份试卷** 作为阶段性估算值；
- 折算单份试卷平均模型成本：**约 0.19~0.20 元 / 份**。

- 项目正式确定供应商及服务方案后，应依据对应正式价格重新测算；

### 5.9 免费额度与 Coding Plan 额度使用边界及风险

#### 5.9.1 免费额度的使用边界与风险
各平台提供的免费额度、赠送额度或新用户优惠，原则上属于试用或促销性质资源，可用于前期验证、功能联调和阶段性成本测算，但不应作为正式运行期的长期预算前提。

当前已调研的国内主流平台公开规则显示，免费额度或新用户优惠通常具有以下共性特征：

- 仅在特定区域、特定账户资格或特定服务范围内有效；
- 普遍存在 **30~90 天** 左右的有效期或 **1 个月左右** 的促销周期；
- 主要覆盖标准实时推理费用，不覆盖批量处理、上下文缓存、模型调优、模型部署或定制化能力相关费用。

基于上述规则，免费额度用于本项目时的风险主要表现为：

1. **服务连续性风险**：免费额度存在有效期、适用范围和额度上限，资源到期、耗尽或平台政策调整后，项目将无法继续按原预算运行；
2. **预算失真风险**：若以免费额度覆盖开发中后期或试运行阶段，会低估正式上线后的真实成本；
3. **自动扣费风险**：部分平台在免费额度耗尽后，若未开启用量停止或预算控制机制，后续调用将自动转为按量计费，并可能导致账户扣费或欠费；
4. **适用范围风险**：免费额度通常只覆盖标准实时推理，不覆盖缓存、批量处理、微调、部署等扩展能力，因此不适合作为完整项目成本模型。

就现有公开证据而言，免费额度用于本项目的主要风险应认定为**服务连续性风险、预算管理风险及合同稳定性风险**，不宜直接表述为一般意义上的法律禁止。

因此，免费额度在本项目中的定位应明确为：

> **仅可作为开发测试、功能验证及阶段性估算参考，不作为正式运行期预算承诺依据。**

#### 5.9.2 Coding Plan 额度的使用边界与风险
Coding Plan 本质上是面向 AI 编程工具的订阅型资源，更适合代码生成、调试、重构与开发提效场景，不应默认等同于可供业务系统直接调用的正式模型服务能力。

当前已调研的国内主流 Coding Plan 产品公开规则显示：

- 部分平台已明确限定其额度**仅在 AI 编程工具场景中生效，不可作为普通业务系统 API 调用资源**；
- 其余平台虽未一律以绝对禁止方式表述，但公开页面与帮助文档均将其定位于 AI 编程工具接入场景，并按编程请求额度或时间窗口进行管理；
- 该类资源整体上更适用于开发辅助与工程提效，而非正式业务运行期的推理入口。

基于上述规则，Coding Plan 额度用于本项目时的风险主要表现为：

1. **平台规则违约风险**：对于已明确限制"不可用于 API 调用"的平台，如将 Coding Plan 额度接入本项目运行时后端，存在违反平台规则的风险；
2. **账号停用风险**：一旦被识别为非编程工具调用、批量化调用或异常调用特征，可能触发限流、订阅失效或账号封禁；
3. **合同适配风险**：即使公开页面未直接表述为"法律禁止"，其产品定位仍明显偏向开发辅助工具，若用于正式业务运行，将存在用途不匹配与合同解释风险；
4. **运行稳定性风险**：Coding Plan 的额度体系通常围绕"请求次数/时间窗口"设计，更适合交互式开发，不适合作为高稳定性、可追踪、可审计的生产运行预算基础。

因此，Coding Plan 的风险表述应以**平台规则限制、账号治理风险与合同适配风险**为主；对于缺乏明确禁止条款、仅体现产品定位的平台，不宜直接上升为一般意义上的法律禁止。

因此，Coding Plan 额度在本项目中的定位应明确为：

> **可用于开发阶段的代码辅助与工程提效，不可作为本项目正式运行期的后端模型调用预算基础。**

#### 5.9.3 本项目的合规预算结论
综合预算测算与外部资源使用边界，本项目应采用如下原则：

- 正式业务运行预算以**商业 API 正式计费**为准；
- 免费额度、赠送额度仅用于前期测试与验证；
- Coding Plan 仅用于开发提效，不纳入业务运行成本模型；
- 任何存在明确使用范围限制的平台资源，均不得作为项目运行时推理入口。
